1. Cold Start & Initialisation
AppNavigator resolves app state before rendering any screen

Cold start sequence

flowchart TD

B --> C[isInitialized = null\nRender loading spinner]

AppContext state flags

  • isInitialized — null while checking, then true/false
  • hasCompletedOnboarding — persisted in OnboardingStorage
  • postOnboardingTab — local to AppNavigator, set by Screen 6 choice NEW

VaultKeyStore (new module)

Previously KeyManager owned the in-memory key cache, causing circular imports with EncryptedStorageService and DocumentRepository. The cache logic now lives in VaultKeyStore.ts which has zero imports from the rest of the app. Both storage modules import getVaultKey() from there directly.

Was: "Create a Family Kit — The only way your loved ones can access this vault. Add documents first, then generate your kit."

Now: "I'll create a Family Kit — Your family's only way in. Set up after you've added your first document."

Safety net banner CHANGED

Was: a single tappable card that opened only the Kit Creation Wizard. Users wanting cloud backup had to navigate to Settings.

Now: a non-tappable card with two side-by-side buttons. "📦 Family Kit" opens the wizard. "☁️ Cloud Backup" calls BackupService.enableCloudBackup() directly, checking availability first and showing a friendly alert if iCloud/Google Drive is unavailable.

Progress ring target CHANGED

Was: always showed X of 24 key documents — a 0% ring on a fresh vault felt discouraging and set an unreachable-looking goal for free users.

Now: non-premium users see X of 5 · free plan. Premium users see X of 24 key documents. The target is achievable, honest, and contextual.

Kit auto-open logic (unchanged)

When showFamilyKitCreationImmediately is true (set by Screen 6 kit choice), the dashboard dismisses the prompt and opens KitCreationWizard — but only if totalDocuments > 0. If the vault is empty the banner guides the user to add docs first. Now that kit-path users land on Documents tab, the vault will typically have at least one doc before they navigate back to Dashboard.

Gate centralised CHANGED

Was: FamilyKitTab owned the gate (checked isPremium, showed PaywallScreen). SettingsScreen had its own gate in FamilyKitSection. Dashboard opened wizard directly (no gate). Three inconsistent behaviours.

Now: KitCreationWizard checks isPremium itself. If not premium it renders PaywallScreen in its place. When the user purchases, isPremium updates in PurchaseContext and the wizard renders on the next cycle. All callers just set visible=true.

FamilyKitTab simplified CHANGED

Was 39 lines with isPremium check, PaywallScreen, and PaywallTrigger state. Now 22 lines — just renders KitHistoryScreen and KitCreationWizard.

11. Dependency Architecture Fix — VaultKeyStore
Require cycles eliminated; runtime stability improved

What VaultKeyStore owns

  • In-memory vault key cache (cachedVaultKey)
  • AppState listener (evicts cache on background)
  • Deduplication of concurrent biometric prompts (pendingKeyPromise)
  • getVaultKey() — SecureStore retrieval with biometric prompt
  • clearCachedKey() — zero-fill + null
  • setCachedKey() — used by KeyManager after key rotation

What KeyManager retains

  • initializeKeys() — first-run vault key generation
  • rotateKeys() — two-phase commit rotation
  • recoverFromInterruptedRotation() — crash recovery
  • restoreFromCloudKeychain() — device migration
  • setBiometricProtection()
  • resetKeys()
